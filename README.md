# Central Leads Management System

A Laravel-based application for collecting contact form submissions from multiple external websites through a centralized API and managing them from a Filament admin dashboard.

## Features

* Centralized lead collection API
* Support for multiple websites and forms
* Filament administration dashboard
* Arabic and English interfaces
* Lead status tracking and management
* Email notifications
* API submission logging
* Dashboard analytics and reporting
* Secure API key generation and storage
* MySQL and SQLite support
* cPanel deployment support

## Requirements

* PHP 8.2 or later
* Composer
* Node.js and npm
* MySQL or SQLite
* A Laravel-compatible web server

## Local Installation

Install the PHP and frontend dependencies:

```bash
composer install
npm install
npm run build
```

Create the local environment file.

Linux, macOS, or Git Bash:

```bash
cp .env.example .env
```

Windows PowerShell:

```powershell
Copy-Item .env.example .env
```

Generate the Laravel application key:

```bash
php artisan key:generate
```

Run the database migrations and local seeders:

```bash
php artisan migrate --seed
```

Start the local development server:

```bash
php artisan serve
```

The application will be available at:

```text
http://127.0.0.1:8000
```

## Admin Dashboard

Open the administration dashboard at:

```text
http://127.0.0.1:8000/admin
```

The dashboard supports Arabic and English. Arabic is the default language, and authenticated users can switch languages from the user menu.

Direct language routes:

* Arabic: `/language/ar`
* English: `/language/en`

## Database Configuration

SQLite may be used for quick local testing.

To use MySQL, update the database settings inside your local `.env` file:

```env
DB_CONNECTION=mysql
DB_HOST=127.0.0.1
DB_PORT=3306
DB_DATABASE=central_leads
DB_USERNAME=root
DB_PASSWORD=
```

Never commit the real `.env` file to the repository.

## Lead Submission API

### Endpoint

```http
POST /api/leads
```

### Authorization

Send the website API key as a Bearer token:

```http
Authorization: Bearer SITE_API_KEY
Content-Type: application/json
Accept: application/json
```

Alternatively, use the `X-API-Key` header:

```http
X-API-Key: SITE_API_KEY
Content-Type: application/json
Accept: application/json
```

### Example Request

```json
{
  "form_key": "quote_home",
  "form_name": "Quote Form",
  "form_type": "quote",
  "name": "Ahmed",
  "email": "ahmed@example.com",
  "phone": "+966500000000",
  "message": "I need a quote for a company website.",
  "page_url": "https://example.com/services",
  "raw_data": {
    "service": "Website Design",
    "budget": "5000 SAR"
  }
}
```

## API Key Security

A local demo API key may be generated by the database seeder:

```text
sk_demo_123456789
```

> This key is intended exclusively for local testing and must never be used in production.

New website API keys are displayed only once when they are created or regenerated.

The complete API key is not stored directly in the database. The system stores:

* A secure hash of the API key
* A short preview for identification

Each production website should use its own unique API key.

## Production Configuration

A production environment template is available in:

```text
.env.cpanel.example
```

Set the production application URL privately on the server:

```env
APP_URL=https://example.com
```

The example environment files must contain placeholders only.

Never include real company domains, passwords, API keys, database credentials, SMTP credentials, or other private information in a public repository.

## Email Configuration

Configure SMTP values privately inside the production `.env` file.

After configuring SMTP, email delivery can be tested with:

```bash
php artisan app:send-test-mail your-email@example.com
```

Never add real SMTP credentials to `.env.example` or `.env.cpanel.example`.

## Production Security Checklist

Before deploying the application:

* Change or remove all demo credentials
* Generate unique API keys for every website
* Set `APP_ENV=production`
* Set `APP_DEBUG=false`
* Use HTTPS
* Configure secure database credentials
* Configure SMTP privately
* Protect the administration dashboard
* Restrict access to the server `.env` file
* Review application and API submission logs
* Do not run unreviewed demo seeders in production
* Never commit secrets to GitHub
