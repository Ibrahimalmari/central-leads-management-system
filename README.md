# Central Leads Management System

A Laravel-based lead management system for collecting contact form submissions from multiple external websites through a centralized API and managing them from a Filament admin dashboard.

## Features

* Centralized lead collection API
* Multiple websites and forms support
* Filament administration dashboard
* Arabic and English interface
* Lead status tracking
* Email notifications
* API submission logs
* Analytics and reporting
* Secure API key management
* MySQL production support
* cPanel deployment support

## Requirements

* PHP 8.2 or later
* Composer
* Node.js and npm
* MySQL or SQLite
* Laravel-compatible web server

## Local Installation

Clone the repository and install the required dependencies:

```bash
composer install
npm install
npm run build
```

Create the local environment file:

```bash
cp .env.example .env
```

Generate the application key:

```bash
php artisan key:generate
```

Run the database migrations and seeders:

```bash
php artisan migrate --seed
```

Start the local development server:

```bash
php artisan serve
```

The application will be available at:

```text
http://127.0.0.1:8000
```

## Admin Dashboard

Open the administration dashboard at:

```text
http://127.0.0.1:8000/admin
```

The dashboard supports Arabic and English.

Arabic is the default language. Authenticated users can switch the interface language from the user menu.

Direct language routes:

* Arabic: `/language/ar`
* English: `/language/en`

## Database Configuration

SQLite may be used for quick local testing.

For MySQL, update the database settings inside your local `.env` file:

```env
DB_CONNECTION=mysql
DB_HOST=127.0.0.1
DB_PORT=3306
DB_DATABASE=central_leads
DB_USERNAME=root
DB_PASSWORD=
```

Never commit the real `.env` file to GitHub.

## Lead Submission API

### Endpoint

```http
POST /api/leads
```

### Authorization

Send the website API key as a Bearer token:

```http
Authorization: Bearer SITE_API_KEY
Content-Type: application/json
Accept: application/json
```

Alternatively, use the `X-API-Key` header:

```http
X-API-Key: SITE_API_KEY
Content-Type: application/json
Accept: application/json
```

### Example Request Body

```json
{
  "form_key": "quote_home",
  "form_name": "Quote Form",
  "form_type": "quote",
  "name": "Ahmed",
  "email": "ahmed@example.com",
  "phone": "+966500000000",
  "message": "I need a quote for a company website.",
  "page_url": "https://example.com/services",
  "raw_data": {
    "service": "Website Design",
    "budget": "5000 SAR"
  }
}
```

## API Key Security

A local demo API key may be generated by the database seeder:

```text
sk_demo_123456789
```

> This key is intended only for local testing. Do not use it on a production website.

New website API keys are displayed only once when they are created or regenerated.

The complete API key is not stored directly in the database. Instead, the system stores:

* A secure hash of the API key
* A short preview for identification

Each production website should have its own unique API key.

## Production Deployment

For cPanel deployment instructions, see:

```text
docs/cpanel-deployment.md
```

A safe production environment template is available at:

```text
.env.cpanel.example
```

Configure the production application URL using your own domain:

```env
APP_URL=https://leads.example.com
```

Replace `https://leads.example.com` with the actual deployment URL privately on the server.

Do not place confidential company domains, passwords, API keys, database credentials, or SMTP credentials in the public repository.

## Email Configuration

Configure SMTP values privately inside the production `.env` file.

After configuring SMTP, test email delivery with:

```bash
php artisan app:send-test-mail your-email@example.com
```

Do not add real SMTP credentials to `.env.example` or `.env.cpanel.example`.

## Security Notes

Before production deployment:

* Change all demo credentials
* Generate unique API keys for every website
* Set `APP_DEBUG=false`
* Use HTTPS
* Configure secure database credentials
* Configure SMTP privately
* Restrict the administration dashboard
* Protect the server `.env` file
* Review application logs and API submission logs
* Never commit secrets or production credentials to GitHub
